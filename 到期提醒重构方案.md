# 到期提醒重构方案

## 1. 目标

把当前小程序里的到期提醒能力，从“页面临时查询 + 零散订阅消息”升级成正式版可上线的提醒系统。

目标效果：

- 企业端能稳定看到站内到期提醒
- 管理端能集中查看重点风险企业
- 微信订阅消息按企业聚合发送，不重复、不骚扰
- 服务端具备幂等、日志、失败排查能力
- 到期状态口径统一，企业端、管理端、AI 问答保持一致

---

## 2. 当前问题概览

### 2.1 触发位置不合理

当前企业端到期提醒主要挂在旧录入页：

- `miniprogram/pages/camera/camera.js`

这意味着用户只有进入录入页时才会触发提醒检查。现在主入口已经变成：

- AI 管家
- 设备中心
- 我的

所以大量企业用户平时不会进入旧录入页，提醒触达不稳定。

### 2.2 去重依赖本地缓存

当前使用：

- `miniprogram/services/expiry-reminder-service.js`

里的 `lastReminderDate` 本地缓存做去重。

这会带来几个问题：

- 换手机、清缓存、重新登录后去重失效
- 同一天内如果已有提醒，新产生的到期数据也不会再次提醒
- 无法服务端统计提醒是否发过

### 2.3 自动推送是“按记录逐条发”

当前云函数：

- `cloudfunctions/expiryReminder/index.js`

里的 `autoScanAndAlert()` 是按记录逐条发送订阅消息。

问题：

- 同一家企业可能收到多条重复消息
- 没有服务端幂等控制
- 没有统一提醒日志

### 2.4 状态口径不统一

当前不同页面和云函数里，到期判断使用了不同口径：

- 有的直接按 `expiryDate`
- 有的按 `status: valid`
- 有的会把 `status` 更新为 `expired`

这会导致：

- 设备中心数字和弹窗提醒对不上
- 管理端风险企业和企业端看到的数据对不上
- AI 管家到期查询结果不稳定

---

## 3. 重构后的总体架构

建议把到期提醒重构成三层：

### 3.1 企业端

- 设备中心展示站内提醒
- 我的页管理提醒设置
- AI 管家支持自然语言查询到期情况

### 3.2 管理端

- 监管首页展示风险总览
- 重点企业列表展示最紧急企业
- 企业风险详情支持继续跟进

### 3.3 服务端

- 统一计算到期状态
- 按企业聚合提醒
- 统一订阅状态管理
- 写入提醒日志
- 通过定时任务执行正式推送

## 4. 数据结构最终版

## 4.1 集合：`expiry_alert_logs`

用途：

- 服务端幂等
- 提醒发送日志
- 失败排查
- 运营统计

建议字段：

```json
{
  "_id": "auto",
  "enterpriseId": "ent_xxx",
  "enterpriseName": "某某企业",
  "district": "大峃所",
  "openid": "oiw_xxx",
  "channel": "wx_subscribe",
  "alertType": "daily_summary",
  "alertDate": "2026-04-20",
  "expiredCount": 3,
  "expiringCount": 5,
  "recordIds": ["rec_1", "rec_2", "rec_3"],
  "templateId": "TEMPLATE_ID",
  "page": "pages/archive/archive?filter=expiring",
  "message": "您有3台压力表已过期，5台将在30天内到期，请及时处理。",
  "status": "success",
  "errorCode": "",
  "errorMessage": "",
  "dedupeKey": "ent_xxx_2026-04-20_daily_summary_wx_subscribe",
  "createdAt": "2026-04-20T09:00:00.000Z",
  "updatedAt": "2026-04-20T09:00:00.000Z",
  "createdBy": "system"
}
```

字段规则：

- `channel`：`wx_subscribe / in_app / sms`
- `alertType`：`daily_summary / expired_alert / expiring_7 / expiring_3 / expiring_1`
- `status`：`pending / success / failed / skipped`
- `dedupeKey`：正式版建议唯一

建议索引：

- `dedupeKey ASC`
- `enterpriseId ASC + alertDate ASC + channel ASC + alertType ASC`
- `createdAt DESC`

## 4.3 现有集合：`pressure_records`

建议补充字段：

```json
{
  "recordStatus": "valid",
  "expiryStatus": "normal",
  "expiryDate": "2026-05-18",
  "lastAlertDate": "2026-04-20",
  "lastAlertType": "daily_summary"
}
```

建议规则：

- `recordStatus`：记录本身是否有效，`valid / void`
- `expiryStatus`：到期状态，`normal / expiring / expired`

说明：

- 不建议继续用单一 `status` 同时表示“记录有效性”和“到期状态”
- 第一阶段如果不想迁历史数据，也可以先由云函数实时计算 `expiryStatus`

## 5.6 定时任务入口

### `runDailyExpiryDigest`

用途：

- 正式版定时任务主入口

入参：

```json
{
  "action": "runDailyExpiryDigest",
  "payload": {
    "days": 30,
    "force": false
  }
}
```

建议执行流程：

1. 查询所有企业到期数据
2. 按企业聚合
3. 查询企业提醒设置
4. 判断今天是否已发送过
5. 发送微信订阅消息
6. 写入 `expiry_alert_logs`

## 9. 提醒策略建议

第一期建议：

- 企业端站内提醒：实时展示
- 微信订阅消息：每天早上 9 点汇总一次
- 管理端风险企业：实时统计
- 同一家企业同一天只发 1 条汇总提醒

后续可扩展：

- 已过期当天额外提醒
- 7天前、3天前、1天前分级提醒
- 短信提醒

## 12. 一句话结论

正式版最合理的到期提醒形态是：

- 企业端：站内实时看
- 微信：按企业聚合每天发一次
- 管理端：集中看重点企业
- 服务端：有日志、有幂等、有订阅配置

这是当前项目最稳、最容易上线、后续也最好扩展的方案。
